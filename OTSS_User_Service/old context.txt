The core idea of the application is - an online support system where users can create tickets and ask for help on something. (categories can be hardcoded)  
Outline what each service will do , the endpoints we will need. And then databases and other technologies we will have to use.

**System Overview**

- User service handles registration, authentication, profile updates, and role assignment through WSO2 IAM 7.2.
- Ticket service manages ticket CRUD, category enforcement, and status transitions; integrates with Queue to track assignment and with Notification for alerts.
- Queue service stores and prioritizes agent assignments, exposes hooks for Ticket service updates, and pushes job events to Notification.
- Notification service delivers email/SMS/in-app alerts for ticket lifecycle events, using queue events and ticket updates.
- WSO2 API Manager fronts all services with throttling, analytics, and policy enforcement; identity flows pass through the IAM for OAuth2/OpenID Connect.

**User Service**

- Endpoints: `/auth/register`, `/auth/login`, `/auth/logout`, `/users/me`, `/users/{id}` (GET/PUT), `/roles` (list/assign).
- Responsibilities: password/SSO flows via IAM, token validation, basic profile data, role-based scopes.
- Storage: PostgreSQL (users, roles, sessions), Redis for session cache, WSO2 IAM directory backing identities.

**Ticket Service**

- Endpoints: `/tickets` (POST/GET), `/tickets/{id}` (GET/PATCH/DELETE), `/tickets/{id}/comments`, `/tickets/{id}/status`, `/categories`.
- Responsibilities: create tickets with hardcoded categories (e.g., IT, Billing, Access), track statuses (Open, In Progress, Resolved, Closed), store comments, emit events to Queue when tickets need assignment, emit events to Notification on lifecycle changes.
- Storage: PostgreSQL (tickets, comments, audit trail).

**Queue Service**

- Endpoints: `/queues` (list available queues), `/queues/{name}/tickets` (GET), `/queues/{name}/tickets/{ticketId}` (assign/update), `/agents/{id}/capacity`.
- Responsibilities: maintain per-category queues, compute priorities (FIFO + SLA), assign tickets to agents, update Ticket service via internal events, push escalation events to Notification.
- Storage: Redis (fast queue ops) plus PostgreSQL for historical assignment data.

**Notification Service**

- Endpoints: `/notifications/test`, `/notifications/email`, `/notifications/sms`, `/subscriptions` (manage per-user preferences).
- Responsibilities: subscribe to Ticket/Queue events through message broker, send email/SMS/webhook notifications, respect user preferences, provide delivery status.
- Tech: message broker (RabbitMQ or NATS), external email/SMS providers (e.g., SendGrid, Twilio), PostgreSQL for notification logs/preferences.

**Cross-cutting Technologies**

- Message broker (RabbitMQ/NATS) for inter-service events.
- API gateway & security: WSO2 API Manager for rate limiting, analytics, mutual TLS; WSO2 IAM 7.2 for OAuth2/OpenID, RBAC, SSO.
- Monitoring/logging: Prometheus + Grafana, ELK stack.
- CI/CD: GitHub Actions.
- Containerization: Docker + Kubernetes for deployment.
